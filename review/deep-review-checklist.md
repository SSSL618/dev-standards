# 深度验收 Checklist

> 起草：质检员 @ 2026-05-05
> 更新：程序员先生 @ 2026-05-05
> 状态：草案
> 用途：用于相对成熟、准备上线/交付/长期使用，或高风险项目的深度验收

## 一、前置判断

- [ ] 项目已相对成熟，不是每天大改阶段
- [ ] 已明确本次是深度验收，不是首轮筛查
- [ ] 已明确项目类型：企业/团队项目 or 个人自用工具
- [ ] 已明确本次验收范围（仓库 / 分支 / PR / 版本 / 发布对象）

## 二、需求与范围对照

- [ ] 已找到需求依据或等效替代材料
- [ ] 已明确本次承诺交付范围
- [ ] 已识别未完成项 / 替代实现 / 降级实现
- [ ] 已识别是否存在做偏 / 做错 / 少做 / 偷换定义

## 三、文档与交付材料

- [ ] 已按 `acceptance-standard.md` 的企业/个人判定表检查文档要求
- [ ] 已确认维护、部署、回滚或恢复路径足够支撑交付

## 四、功能实现正确性

- [ ] 核心功能完整手测
- [ ] 核心路径已走通
- [ ] 异常路径已检查
- [ ] 空状态 / 边界条件已检查
- [ ] 回归影响已检查

## 五、代码质量审查

- [ ] 已看关键模块，而非只看入口
- [ ] 模块职责清晰
- [ ] 无明显重复逻辑、硬编码、临时拼接、废弃残留
- [ ] 错误处理完整，无危险静默失败

## 六、测试与验证证据

- [ ] 已按 SOP 的证据最小格式表收齐证据
- [ ] 已确认构建、测试、lint / typecheck / CI 状态
- [ ] 已标明不能自动验证部分的限制

## 七、安全 / 隐私 / 权限

- [ ] 未提交 token / secret / 私钥 / 密码
- [ ] 未暴露真实隐私数据
- [ ] 未发现明显未鉴权 / 越权 / 危险默认行为
- [ ] 高权限能力边界清楚

## 八、依赖 / 合规 / 外部资源

- [ ] 关键依赖来源合理
- [ ] 第三方代码 / 资源来源基本清楚
- [ ] 未发现明显 License / 版权 / ToS 风险

## 九、部署 / 运行 / 可维护性

- [ ] 能说明如何运行 / 构建 / 启动
- [ ] 环境依赖明确
- [ ] 部署 / 安装路径清楚
- [ ] 出问题时基本知道如何排查

## 十、验收结论输出

- [ ] 已按 SOP 的结论阈值表输出结论
- [ ] 已按严重程度区分问题
- [ ] 已写清问题 / 原因 / 影响 / 建议
- [ ] 已写清未覆盖 / 未闭环验证项
