# 首轮验收 Checklist

> 起草：质检员 @ 2026-05-05
> 更新：程序员先生 @ 2026-05-05
> 状态：草案
> 用途：用于新项目或未成熟项目的首轮验收 / 风险筛查

## 一、项目定位

- [ ] 已确认这是首轮验收，不是深度验收
- [ ] 已明确项目类型：企业/团队项目 or 个人自用工具
- [ ] 已明确目标：筛风险 / 判断是否值得继续 / 判断是否进入深验

## 二、最小背景信息

- [ ] 仓库 / 项目路径明确
- [ ] 分支 / PR / 当前版本明确
- [ ] 能找到最小范围说明（README / PROJECT / HANDOVER / PRD 任一）
- [ ] 能判断项目当前大致做什么
- [ ] 能判断项目当前是否接近完成

## 三、文档与依据

- [ ] 已按 `acceptance-standard.md` 的企业/个人判定表检查需求依据要求
- [ ] 已确认基础说明材料足够支持首轮判断

## 四、最小验证

- [ ] 能跑的已跑（build / test / CI / 页面 / 接口 / 脚本）
- [ ] 不能跑的已明确标注“未闭环验证”
- [ ] 已按 SOP 的证据最小格式表记录验证证据

## 五、关键代码抽查

- [ ] 看了入口文件
- [ ] 看了核心模块
- [ ] 看了配置 / 环境变量 / 存储相关代码
- [ ] 看了权限 / 密钥 / 外发 / 删除 / 写入等高风险代码

## 六、首轮必须筛出的风险

- [ ] 明显功能偏差
- [ ] 明显未完成却声称完成
- [ ] 密钥 / 隐私数据误提交
- [ ] 明显未鉴权 / 越权 / 危险默认行为
- [ ] 缺少最基本使用说明
- [ ] 明显临时拼接、难维护实现

## 七、首轮结论

- [ ] 已判断是否值得继续投入
- [ ] 已判断是否建议进入深验
- [ ] 已明确本次仅为首轮，不伪装成全量代码已审
