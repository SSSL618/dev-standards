# GitHub 项目验收 Checklist（Reviewer 执行版）

> 起草：质检员 @ 2026-05-05
> 更新：质检员 + 程序员 联合审定 @ 2026-05-05
> 状态：草案（合并版）
> 定位：统一执行入口，不重复定义判定口径
> 引用标准：结论口径引用 `acceptance-standard.md`，证据格式引用 `证据最小格式表`，阈值引用 `严重级别→结论阈值表`

---

## 一、验收对象确认

说明：更新验收的分级与复核深度，统一引用 `tiered-review-standard.md` 第三节“变更类型 → 复核深度表”，不在本清单内重复判断口径。


- [ ] 仓库地址明确
- [ ] 分支 / PR / Tag / Release 范围明确
- [ ] 本次验收类型：首轮验收 / 深度验收 / 更新验收（引用 `tiered-review-standard.md`）
- [ ] 对应 PRD / Issue / 任务单已提供
- [ ] 测试环境 / 演示地址 / 本地运行方式已提供

---

## 二、阻塞项先排查

以下任一缺失，先记 **Blocked**，不继续执行验收：

- [ ] 无需求依据（PRD / Issue / 任务单）或等效替代材料（见 `acceptance-standard.md` 第5节企业/个人判定表）
- [ ] 无启动方式 / 无法运行
- [ ] 无测试证据（引用 `证据最小格式表`）
- [ ] 无法访问关键页面 / 环境 / 仓库权限不足
- [ ] 验收范围不清

---

## 三、执行动作清单

### A. 项目基础完整性

**动作：**
1. 读取 README，验证是否包含：项目用途、启动方式、依赖配置、部署方式、维护路径
2. 检查是否存在 `.env.example` 或等效配置说明
3. 记录缺失项

**输出：**
- 通过项：有 README 且内容完整
- 问题项：[缺少的具体要素]
- 引用标准：`acceptance-standard.md` 第六节 A

---

### B. 产品与需求一致性

**动作：**
1. 读取 PRD / Issue / 任务单，明确本次承诺范围
2. 逐项核对核心功能实现
3. 检查是否存在：删减、替代实现、降级实现、未说明的功能变化
4. 重点验证核心路径（注册/登录/创建/编辑/删除/支付/导出/同步等）

**输出：**
- 已验证项：[核心功能名称]
- 问题项：[不一致 / 缺失的具体项]
- 引用标准：`acceptance-standard.md` 第六节 B

---

### C. 文档完整性

**动作：**
1. 按 `acceptance-standard.md` 第六节 C 逐项检查
2. 区分企业项目 vs 个人工具（引用判定表）
3. 记录缺失的必需项 vs 可替代项

**输出：**
- 通过项 / 缺失项 / 可替代项（个人工具）
- 引用标准：`acceptance-standard.md` 第五节 企业/个人判定表

---

### D. 代码质量与可维护性

**动作：**
1. 读取关键模块代码（入口文件、核心模块、高风险代码）
2. 验证：模块边界清晰、无明显重复逻辑、命名清晰、错误处理完整、配置解耦、无废弃残留
3. 执行 lint / typecheck / build / test（如适用），记录结果

**输出：**
- 通过项：[关键模块名] 通过代码质量检查
- 问题项：[具体问题与位置]
- 引用标准：`acceptance-standard.md` 第六节 D

---

### E. 测试与稳定性

**动作：**
1. 收集测试证据，对照 `证据最小格式表`（企业五要素，个人三要素）
2. 验证：核心路径、异常路径、边界条件、回归影响
3. 记录：覆盖范围与未覆盖范围

**输出：**
- 已验证项：[具体功能与验证方式]
- 证据清单：[截图/日志/CI链接]
- 未覆盖项：[明确说明未验证的范围与原因]
- 引用标准：`证据最小格式表`

---

### F. 安全、隐私与密钥泄露

**动作：**
1. 搜索仓库是否包含：token / secret / API key / 私钥 / 账号密码 / Cookie / session
2. 检查日志、fixture、截图、测试数据是否泄露真实隐私
3. 检查 `.env` / 证书 / 备份文件是否误提交
4. 验证权限控制：是否存在未鉴权接口 / 越权访问
5. 检查依赖是否有已知高危漏洞

**输出：**
- 通过项 / 问题项（列具体位置与风险）
- 引用标准：`acceptance-standard.md` 第六节 F

---

### G. 法律、版权与合规

**动作：**
1. 检查第三方代码 / 素材来源是否清晰
2. 验证 License 与项目用途兼容
3. 检查是否存在受版权保护但未标注来源的内容
4. 验证涉及隐私收集 / AI 内容 / 外部素材是否有说明

**输出：**
- 通过项 / 问题项
- 引用标准：`acceptance-standard.md` 第六节 G

---

### H. 部署、运维与可回滚

**动作：**
1. 验证部署文档是否可执行
2. 检查环境变量、数据库迁移、构建参数是否清晰
3. 验证是否有日志 / 健康检查 / 错误监控（至少一种）
4. 检查是否有回滚方案和数据备份说明

**输出：**
- 通过项 / 问题项
- 引用标准：`acceptance-standard.md` 第六节 H

---

### I. GitHub 协作质量

**动作：**
1. 检查 PR 描述是否完整（引用 `submission-standard.md` PR 模板）
2. 验证 commit 是否可读、是否混入无关改动
3. 检查是否关联 Issue / PRD
4. 验证 review 意见是否已处理或说明未处理原因
5. 检查是否误提交构建产物 / 临时文件 / 个人配置

**输出：**
- 通过项 / 问题项
- 引用标准：`submission-standard.md`

---

## 四、结论输出

### 结论判定（引用 `严重级别→结论阈值表`）

- [ ] **通过（Pass）**
- [ ] **有条件通过（Pass with Conditions）**（附：负责人 + 截止时间 + 临时缓解方案）
- [ ] **不通过（Fail）**
- [ ] **阻塞（Blocked）**

### 问题清单

对每个问题，必须输出：
- **问题**：具体位置与现象
- **原因**：对照哪条标准
- **影响**：影响范围（功能/用户/数据/上线风险）
- **建议**：怎么修
- **严重级别**：P0 / P1 / P2 / P3

### 已验证通过项

只列关键通过项，不写流水账。

### 待补材料 / 待确认项

记录阻塞项与不确定项。

---

## 五、执行约束

1. **不重复定义判定口径**：结论判定统一引用 `acceptance-standard.md` 中的 `严重级别→结论阈值表`
2. **不跳过高风险模块**：即使其他项通过，发现安全/隐私问题仍需记录
3. **证据不足先记阻塞**：不因项目紧急而降低证据要求
4. **区分企业 vs 个人**：按企业/个人判定表确定检查范围，不将企业标准强压到个人工具
