# 项目验收分层标准（首轮验收 / 深度验收 / 更新验收）

> 起草：质检员 @ 2026-05-05
> 更新：程序员先生 @ 2026-05-05
> 状态：草案
> 用途：统一规定什么时候做首轮验收、什么时候做深度验收、项目更新后如何做更新验收

## 一、目标

把“验收”拆成三层，避免尺度混乱：
1. **首轮验收**：快速判断项目是否达到下一阶段门槛
2. **深度验收**：对相对成熟项目做更完整的质量审查
3. **更新验收**：项目已有过往验收结论后，对新增改动做增量复核

## 二、什么时候做哪一层

### 1. 首轮验收
适用：
- 新项目第一次进入验收视野
- 项目还在快速开发，完成度不明
- 需要先筛出明显风险，再决定是否深验
- 同时有多个项目排队，需要先做优先级筛查

### 2. 深度验收
适用：
- 项目已接近完成
- 准备上线 / 交付 / 发布 / 长期自用
- 首轮结论显示项目已基本成熟
- 涉及隐私、权限、高权限、本地写操作、支付、部署、外部 API 成本等高风险内容

### 3. 更新验收
适用：
- 项目已有历史验收结论
- 本次是增量改动，不是全新验收
- 需要根据改动范围决定轻验、标准复核、局部深验或完整深验

## 三、更新验收：变更类型 → 复核深度表

| 变更类型 | 建议复核深度 |
|---|---|
| 文案修改、样式微调、注释调整、非关键 README 更新 | 轻量更新验收 |
| 非核心配置微调 | 轻量更新验收 |
| 新增中小功能、修改核心流程的一部分 | 标准更新验收 |
| 修改依赖、局部数据结构、局部权限逻辑 | 标准更新验收 |
| 改鉴权、支付、存储、部署 | 局部深验 或 完整深验 |
| 改 schema / migration / 核心数据模型 | 局部深验，并强制检查兼容与回滚 |
| 重构核心架构、更换存储层 / 鉴权层 / 部署方式 | 完整深验 |
| 引入新的外发链路、高权限能力、敏感数据处理方式 | 完整深验 |

规则：
- 若同一次改动同时命中多个类型，按**最高复核深度**执行
- 若改动已接近“半个新项目”，直接按完整深验处理
- 更新验收的分级判断，以本表为唯一引用口径

## 四、旧结论何时失效

出现以下任一情况，旧结论默认部分或全部失效：
1. 核心架构重构
2. 部署链路改变
3. 数据存储方式改变
4. 鉴权 / 权限模型改变
5. 引入新的第三方模型 / API / 外发链路
6. 从个人工具变成团队工具 / 对外工具
7. 功能范围大幅扩张
8. 上次问题的整改方式改动了核心实现

## 五、默认决策规则

- **新项目 + 未成熟** → 首轮验收
- **新项目 + 已接近完成 / 高风险** → 深度验收
- **已验收项目 + 小改 / 中改** → 更新验收
- **已验收项目 + 大改 / 重构** → 重新深度验收

## 六、一句话规则

- 新项目：先首轮，再决定要不要深验
- 成熟项目：准备交付 / 上线 / 长期用时做深验
- 已验收项目的后续修改：先判变更类型，再决定复核深度
