# SOP：GitHub 项目验收与交付 Review 标准

> 起草：质检员 @ 2026-05-05
> 更新：程序员先生 @ 2026-05-05
> 状态：草案
> 触发条件：需要对 GitHub 上的项目、仓库、版本迭代、PR 或阶段性交付进行验收
> 适用环境：GitHub 仓库 / 产品型项目 / 内部工具项目 / Web 与自动化项目

## 一、目标

建立一套统一、可执行、可复核的项目验收标准，用于工程师自检或 reviewer 验收，覆盖以下维度：
- 产品功能与 PRD 一致性
- 文档完整性
- 代码质量与可维护性
- 测试与稳定性
- 隐私、安全与密钥泄露风险
- 法律、版权与合规风险
- 部署、运维与交付可持续性

## 二、验收结论

验收结论只允许以下 4 种：
- **通过（Pass）**
- **有条件通过（Pass with Conditions）**
- **不通过（Fail）**
- **阻塞（Blocked）**

## 三、问题严重级别

- **P0 / 阻断**：功能核心错误、数据错误、严重安全漏洞、密钥泄露、法律高风险、无法部署、无法回滚
- **P1 / 高**：主要功能不符合 PRD、关键路径体验异常、测试明显不足、权限控制错误、隐私边界不清
- **P2 / 中**：文档缺漏、错误处理不足、监控缺失、代码可维护性较差、边界条件覆盖不足
- **P3 / 低**：命名、注释、格式、轻微文案、非关键体验问题

## 四、验收前置条件

开始验收前，至少确认以下材料齐备：
1. 仓库地址、分支/PR/版本号明确
2. 对应 PRD、需求文档、任务单或验收范围明确
3. 本次交付清单明确
4. 测试方式与测试证据可获取
5. 部署地址、演示环境或运行方式可获取
6. 如涉及账号、支付、隐私数据、第三方 API，需有说明文档

若以上任一缺失，先记为 **阻塞项**。

## 五、三张基础判定表（强制件）

### 1. 证据最小格式表

| 项目 | 企业/团队项目 | 个人自用工具 |
|---|---|---|
| 验证环境 | 必需 | 必需 |
| 验证步骤 | 必需 | 必需 |
| 实际结果 | 必需 | 必需 |
| 截图 / 日志 / CI 链接 | 必需 | 可降级 |
| 未覆盖范围 | 必需 | 必需 |

规则：
- 个人工具若缺截图 / 日志 / CI 链接，必须明确说明原因，且不能影响关键结论可追溯性
- 涉及支付、权限、数据写入、部署、迁移等高风险改动时，个人工具也不建议降级证据

### 2. 严重级别 → 结论阈值表

| 条件 | 默认结论 |
|---|---|
| P0 >= 1 | 不通过 |
| P1 >= 1，且影响核心路径 / 数据正确性 / 安全 / 隐私 / 发布 | 不通过 |
| P1 >= 1，但可临时缓解且不阻断交付 | 有条件通过 |
| 仅有 P2 / P3 | 通过 或 有条件通过 |
| 缺少关键输入，无法判断 | 阻塞 |

P1 被判为“有条件通过”时，必须同时满足：
1. 有负责人
2. 有截止时间
3. 有临时缓解方案
4. 不影响当前核心交付成立

### 3. 企业项目 vs 个人工具判定表

| 检查项 | 企业/团队项目 | 个人自用工具 |
|---|---|---|
| 需求依据（PRD / Issue / 任务单） | 必需 | 可替代（README / HANDOVER / 项目说明） |
| 自动化测试 / CI | 必需 | 可降级 |
| 截图 / 日志 / CI 证据 | 必需 | 可降级 |
| 部署 / 回滚说明 | 必需 | 可降级，但需有恢复路径 |
| 交接 / 运维文档 | 必需 | 可替代（最小维护说明） |
| 权限 / 外发 / 密钥边界说明 | 必需 | 必需 |
| 隐私 / 安全检查 | 必需 | 必需 |
| 已知限制 / 风险披露 | 必需 | 必需 |

说明：
- **必需**：缺失即不能判定通过
- **可降级**：形式可简化，但不能完全没有
- **可替代**：允许用等效材料替代，不强制企业形态文档

补充说明：
- 更新验收的“变更类型 → 复核深度”映射表，不放在本 SOP 中重复维护，统一以 `tiered-review-standard.md` 为唯一引用口径

## 六、正式验收维度

### A. 项目基础完整性
必查：项目目标、README、启动方式、依赖配置、部署方式、环境变量示例、维护路径。

### B. 产品与需求一致性
必查：需求依据、范围覆盖、核心流程、业务规则、异常状态、边界条件、未说明删减或替代实现。

### C. 文档完整性
必查：需求来源、技术方案或最小设计说明、部署/运维说明、接口说明、测试说明、风险与限制说明。

### D. 代码质量与可维护性
必查：模块边界、重复逻辑、命名、错误处理、日志安全、配置解耦、废弃残留、依赖合理性、lint/typecheck/build/test 状态。

### E. 测试与稳定性
必查：引用本 SOP 的**证据最小格式表**；核心路径、异常路径、边界条件、回归影响、构建与发布流程。

### F. 安全、隐私与密钥泄露
必查：密钥、账号、Cookie、隐私数据、日志脱敏、权限控制、未鉴权/越权、常见注入与上传风险、依赖高危漏洞。

### G. 法律、版权与合规
必查：第三方代码/素材来源、License 兼容性、数据使用边界、AI 内容来源、平台 ToS / API 政策风险。

### H. 部署、运维与可回滚性
必查：部署可复现性、环境变量、迁移说明、日志/告警/健康检查、发布步骤、回滚方案、备份与恢复路径。

### I. GitHub 协作质量
必查：PR 描述质量、commit 可读性、关联需求、review 意见处理状态、误提交产物、分支保护与 CI 规则。

GitHub 协作的具体提交规范，统一引用：`submission-standard.md`

## 七、最低准入门槛（硬门槛）

以下任一不满足，默认不能判定“通过”：
1. 有明确验收范围
2. 有需求依据或等效替代材料
3. 有基础文档（README + 启动/部署说明）
4. 有至少一种可追溯测试证据
5. 无真实密钥 / 隐私数据泄露
6. 核心功能与需求一致
7. 能构建 / 运行 / 演示
8. 关键问题已分级并记录

## 八、建议的验收执行顺序

1. 确认范围
2. 看需求与文档
3. 跑基础验证
4. 核核心功能与异常路径
5. 查高风险代码与数据链路
6. 查安全、隐私、依赖、合规
7. 输出结论与分级问题

## 九、验收报告输出模板

### 1. 结论
- 结论：通过 / 有条件通过 / 不通过 / 阻塞
- 范围：仓库 + 分支/PR + 验收时间
- 依据：需求文档 / 测试证据 / 环境

### 2. 阻断问题（如有）
每条必须包含：
- 问题
- 原因
- 影响
- 建议

### 3. 非阻断问题
按 P1 / P2 / P3 分组列出。

### 4. 已验证通过项
只列关键通过项。

### 5. 待补材料 / 待确认项
用于标记阻塞项与不确定项。

## 十、判断原则

1. 不替项目组脑补“应该没问题”
2. 没证据，不算通过
3. 能跑不等于可交付
4. 文档缺失本身就是交付问题
5. 密钥、隐私、版权、越权问题优先级高于样式问题
6. 若需求本身不可验证，先指出“需求不可验收”
