# Web App 默认部署验收 SOP

状态：试运行
作者：Codex
最后更新：2026-05-05
触发条件：新 Web App 按默认架构完成部署后，需要确认正式域名、页面、API、写入链路、环境变量与回滚入口是否正常
适用环境：`Cloudflare + Vercel + Vercel Functions / API + Firebase / Firestore` 默认链路

## 一、文档定位

本 SOP 默认服务于以下标准链路：

```text
浏览器
  -> Cloudflare
  -> Vercel
  -> Vercel Functions / API
  -> Firebase / Firestore / 第三方服务
```

只要后续 Web App 没有特别说明，部署验收默认按这份 SOP 执行。

这份 SOP 不负责项目搭建，只负责：

- 部署完成后的验收
- 排障起点判断
- 回滚判断

## 二、执行前准备

执行本 SOP 前，先准备：

- 正式访问域名
- 当前最新 deployment URL
- 至少 2 个关键 API 路径
- 至少 1 条关键写入路径
- 当前项目的环境变量清单
- 上一版可用 deployment 或明确回滚入口

## 三、一级验收：确认线上对象对不对

### Step 1：确认正式域名是否指向当前版本

检查：

- 正式域名是否已绑定
- 正式域名是否真的指向最新成功 deployment
- 没有把失败 deployment、旧 deployment、preview deployment 误当生产版本

通过标准：

- 正式域名访问到的是预期版本
- 页面内容与当前代码预期一致

失败时先看：

- Vercel 域名绑定
- deployment 绑定关系
- Cloudflare DNS 与缓存状态

### Step 2：确认 Cloudflare 入口正常

检查：

- DNS 记录是否正确
- 代理状态是否符合预期
- TLS / SSL 是否正常
- 是否存在明显缓存旧内容

通过标准：

- 正式域名可稳定打开
- 没有基础证书错误
- 没有明显的旧版本缓存问题

失败时先看：

- Cloudflare
- 域名解析
- SSL / TLS 设置

## 四、二级验收：确认页面和 API 真可用

### Step 3：页面可用性检查

至少验证：

- 首页正常打开
- 关键入口可见
- 无首屏致命报错
- 无明显静态资源 404

通过标准：

- 用户可以开始核心流程
- 页面不是“能开但立即卡死”

### Step 4：关键 API 检查

至少测试：

- 1 个健康类或基础读接口
- 1 个关键业务读接口
- 1 个关键业务写接口对应的结果读取

建议方式：

- 浏览器实际操作一次
- 必要时用 `curl` 或等价方式验证状态码

通过标准：

- 关键读接口返回预期状态
- 关键写入执行后，后续读取能看到结果

失败时先看：

- Vercel deployment 状态
- Vercel Function 日志
- API 路径和路由结构

## 五、三级验收：确认数据与副作用正确

### Step 5：关键写入链路检查

至少验证一条完整链路：

1. 触发一次关键写入
2. 刷新页面或重新读取
3. 确认结果仍然存在

通过标准：

- 数据成功持久化
- 刷新后状态仍保留
- 不是只改了前端本地状态

失败时先看：

- 服务端写入逻辑
- Firestore 配置与权限
- 是否依赖响应后异步副作用

### Step 6：刷新或实时机制检查

如果项目依赖 Firestore listener 或其他刷新机制，检查：

- 新数据写入后页面能否看到更新
- 如果依赖显式刷新，刷新逻辑是否正常
- 没有把“本地实时表现”误当成“线上一致性已经成立”

通过标准：

- 页面数据更新逻辑符合当前项目设计
- 不存在明显的“写成了但界面不同步”假象

## 六、四级验收：环境变量与部署环境检查

### Step 7：环境变量检查

确认：

- `Production` 与 `Preview` 的变量分层清楚
- 服务端密钥已正确配置在部署平台
- 前端没有暴露不应暴露的高价值密钥
- Firebase 服务端凭据格式正确

通过标准：

- 线上函数可以正常访问依赖服务
- 不存在“本地正常、线上缺变量”的问题

失败时先看：

- Vercel 环境变量
- Firebase 服务端凭据
- 第三方 API key 配置

## 七、排障分层顺序

### A. 域名打不开 / 证书错误 / 页面版本不对

先看：

- Cloudflare
- 域名绑定
- deployment 绑定关系

### B. 页面能打开，但 API 不通

先看：

- Vercel deployment 状态
- Vercel Function 日志
- API 路径和路由结构

### C. API 通了，但写入不保留

先看：

- 服务端写入逻辑
- Firestore 权限与配置
- 是否依赖响应后异步副作用

### D. 本地正常、线上异常

先看：

- 环境变量差异
- 路由差异
- 部署目标是否真是当前代码
- 是否把 preview 当 production

## 八、最低回滚规则

满足以下任一项，应优先考虑回滚，而不是继续在线硬修：

- 首页不可用
- 关键写入链路失效
- 高价值功能完全不可用
- 当前异常短时间内无法定位到平台层还是代码层
- 已确认上一版 deployment 正常可用

回滚前至少确认：

- 上一版可用 deployment 存在
- 回滚不会破坏当前数据结构
- 当前异常不是单纯的 Cloudflare 缓存问题

## 九、部署通过标准

以下全部满足，才算本次部署通过：

- 正式域名绑定正确
- 页面正常打开
- 关键 API 可访问
- 至少一条关键写入链路成功且刷新后保留
- 环境变量无缺失
- 当前版本与用户实际访问版本一致
- 当前回滚入口明确

## 十、维护规则

- 每次因真实部署故障新增了稳定经验，必须回写本 SOP
- 项目私有路径、项目私有变量名、项目私有命令，不写入共享版 SOP
- 如果某条规则只适用于单个项目，应留在项目文档，不应提升为共享基线
