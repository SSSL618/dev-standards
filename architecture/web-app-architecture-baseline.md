# Web App 默认部署架构基线

状态：试运行
作者：Codex
最后更新：2026-05-05
适用对象：后续新建 Web App 项目
定位：默认架构基线，不是可选参考之一

---

## 一、文档定位

从现在开始，后续新建 Web App 项目，默认优先采用以下架构：

```text
浏览器
  -> Cloudflare
  -> Vercel
  -> Vercel Functions / API
  -> Firebase / Firestore / 第三方服务
```

这不是“符合某些条件时可考虑”的备选方案，而是默认起步架构。

如果某个项目要偏离这套架构，必须能明确说明偏离原因，并在项目文档中写清楚：

- 为什么不继续使用默认架构
- 替代架构是什么
- 新架构带来的收益是什么
- 新架构新增了哪些运维成本和风险

---

## 二、默认规则

后续做新的 Web App，默认遵循以下规则：

1. 前端页面默认部署在 `Vercel`
2. 域名和 DNS 默认放在 `Cloudflare`
3. 数据主路径默认使用 `Firebase / Firestore`
4. 所有关键写入和密钥调用默认都走服务端
5. 默认不把 `SQLite` 当生产数据层
6. 默认不把 `WebSocket` 当 Vercel 上的核心同步机制
7. 默认通过 GitHub 承接代码托管和自动部署链路

这套默认规则的目标是：

- 低运维
- 快速上线
- 易于迭代
- 易于排障
- 适合自己和少量朋友长期使用

---

## 三、各层职责

### 1. Cloudflare

默认职责：

- 管理域名 DNS
- 作为统一公网入口
- 提供 TLS / SSL
- 提供基础代理能力
- 提供基础缓存和安全层

默认不承担：

- 项目主业务逻辑
- 核心状态同步
- 主数据库角色

### 2. Vercel

默认职责：

- 托管前端页面
- 托管轻量 API
- 保存部署环境变量
- 承接 GitHub 自动部署
- 作为浏览器与第三方服务之间的服务端中转层

### 3. 服务端 API

默认职责：

- 保存高价值密钥
- 接收浏览器请求
- 调用 Gemini / Firebase / 第三方私密 API
- 做关键数据写入
- 做权限与业务规则校验
- 返回最终结果给浏览器

### 4. Firestore

默认职责：

- 作为主数据层
- 承接结构化读写
- 在需要时提供轻量实时监听能力

---

## 四、固定原则

### 1. 浏览器不直接承担密钥责任

以下请求默认不应由浏览器直连：

- Gemini / OpenAI / Claude 等 AI 请求
- 地图、天气等带 key 的第三方 API
- 图片分析、截图分析、文件解析
- 关键数据写入
- 需要业务规则校验的操作

结论：

- 浏览器只请求自己的正式域名和自己的 API
- 密钥默认只保存在服务端环境变量里
- 关键请求默认由服务端中转

### 2. Firestore 是默认主路径，但不要把浏览器直连当万能解法

可以使用：

- Firestore 作为主数据层
- Firestore listener 作为轻量实时刷新机制

但默认不应：

- 把关键权限和关键写入只留在前端
- 把浏览器直连 Google 服务当成所有核心链路
- 把“前端能写”误当成“架构已经安全可维护”

### 3. Vercel 上不要依赖“响应后异步副作用”

不应默认假设以下模式在线上可靠：

1. 接口先返回 200
2. 再在函数尾部异步执行关键重算 / 持久化 / 刷新

关键副作用默认应满足二选一：

- 在请求生命周期内完成
- 由前端显式发起第二次请求

### 4. 路由设计优先降低部署和排障复杂度

默认优先选择更平的 API 结构，不要在第一版就堆深层复杂动态路由。

默认原因：

- 更容易 smoke test
- 更容易定位 404 和构建冲突
- 更适合快速上线和回归验证

这不是说深层路由不能用，而是默认起步阶段优先保守。

### 5. 不把 WebSocket 当默认核心同步机制

默认优先：

- Firestore listener
- 轮询
- 显式刷新
- 拆分成多次请求的同步机制

默认不优先：

- 把核心一致性依赖在 Vercel 上的 WebSocket 运行假设上

---

## 五、默认部署前检查

上线前默认检查：

- 数据主路径已明确，不再依赖本地 `SQLite`
- 服务端和前端都能正常读写 Firestore
- 已有 seed 或初始化脚本
- 后端能力可运行在 `Vercel Functions / api/*`
- 关键 API 路径没有明显路由冲突
- 环境变量边界清楚
- 至少有一套最小验收路径

---

## 六、默认部署后最低验收要求

部署后至少确认：

- 正式域名已绑定到预期 deployment
- 首页可以打开
- 至少一个关键 GET API 可访问
- 至少一条关键写入链路成功
- 写入后刷新仍能保留
- 环境变量在 `Production` 与 `Preview` 上配置清楚
- 当前版本的回滚入口明确

---

## 七、偏离默认架构的典型条件

以下情况可以偏离默认架构，但必须单独说明：

- 需要中国大陆公众产品级稳定性
- 需要完全脱离 Google 生态
- 需要长连接 / 常驻服务 / 自定义后端运行时
- 需要更重的服务端状态管理
- 需要平台级实时计算或复杂任务编排
- 需要更强的地域、合规或网络控制能力

---

## 八、边界声明

这套默认架构的目标是：

- 尽量降低部署成本
- 尽量降低运维复杂度
- 尽量提升自己和少量朋友的可用性
- 保持轻量开发和快速迭代体验

这套默认架构不承诺：

- 中国大陆生产级稳定访问
- 大规模公众服务能力
- 高合规环境下的正式商用稳定性

如果项目目标升级为公众产品，应重新设计平台与地域架构，而不是继续在默认架构上叠补丁。


## 九、Vertex AI 接入

默认使用 Vertex Express Mode，不使用 ADC。

### 初始化代码

```javascript
new GoogleGenAI({
  vertexai: true,
  apiKey,
  apiVersion: 'v1',
});
```

### 环境变量

```env
GOOGLE_GENAI_USE_VERTEXAI=true
GEMINI_API_KEY=your_google_cloud_api_key
GOOGLE_CLOUD_LOCATION=global
```

### 关键约束

- `project/location` 和 `apiKey` 不能同时传给 SDK
- Express Mode 只需要 `vertexai: true` + `apiKey` + `apiVersion`
- ADC 模式仅在 GCP 原生环境（Firebase Hosting、Cloud Run）使用

详细迁移步骤参见 [Vertex AI 迁移指南](../migrations/vertex-ai-migration.md)。
